[1:00 — ARCHITECTURE]

(show assets/architecture_diagram.png)

Let me walk you through the flow.

It starts with ContextAgent. This agent calls our MCP server to fetch the PR diff,
the list of changed files, and sibling file context — so it understands not just
what changed, but what surrounds it.

From there, SecurityAgent and QualityAgent run in parallel. SecurityAgent calls
run_semgrep and run_sca through the MCP server to detect hardcoded secrets,
vulnerable dependencies, and injection risks. QualityAgent independently audits
for code smells, error handling gaps, and convention violations.

Both agents feed their findings into a JoinNode, which waits for both to finish
before continuing.

Then comes the most critical node — security_checkpoint. This is fully deterministic,
no LLM involved. It reads the findings, checks for any CRITICAL severity item, and
routes the workflow: BLOCK_MERGE if a critical issue exists, AUTO_COMMENT otherwise.
It also writes a structured audit log line to stdout for traceability.

Finally, ReviewComposerAgent takes the decision and all findings, composes a
comprehensive PR review comment with a findings table grouped by severity, and
posts it to GitHub using post_pr_comment. It also calls set_check_run_status to
set the GitHub Check Run — so the PR shows a red or green status block.
